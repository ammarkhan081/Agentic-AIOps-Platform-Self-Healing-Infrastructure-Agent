1. Executive Summary
Mission Statement
Design and deploy a production-grade, multi-agent autonomous platform that continuously monitors a live microservices infrastructure, detects anomalies in real time, reasons about root causes using LLM-based causal analysis, generates and executes remediation actions, validates recovery, and learns from every incident to reduce future mean time to recovery.
The Agentic AIOps Self-Healing Infrastructure Agent (ASHIA) represents a fundamentally different class of AI project. Unlike research assistants or RAG chatbots that produce documents, ASHIA controls live infrastructure. It perceives real system telemetry, reasons about failures, and executes real actions via API calls to the Docker SDK. This closed-loop autonomous operation is what makes it rare, technically demanding, and immediately compelling to any engineering hiring manager.
The system is deliberately scoped to a realistic demo environment â€” a three-service Docker Compose application â€” making it completable in six weeks while demonstrating every principle that would apply at enterprise scale. It is designed to be extended to Kubernetes in v1.1 without architectural changes.

2. Problem Statement
2.1 The Status Quo Is Broken
Modern microservices architectures fail in ways that are impossible for humans to monitor manually. A production system emitting thousands of metric data points per minute across dozens of services requires automated intelligence â€” not dashboards that humans stare at.
Problem 1:
 Detection lag â€” A human on-call engineer typically takes 15â€“30 minutes to notice a degradation that has already been visible in metrics for 5 minutes. Every minute costs money.
Problem 2:
 Diagnosis bottleneck â€” Root cause analysis across distributed traces, logs, and metrics simultaneously is cognitively overwhelming. Engineers focus on the obvious signal and miss cascading causes.
Problem 3:
 Runbook rigidity â€” Standard runbooks handle known failure modes. Novel failures â€” a new memory leak pattern, an unexpected DB lock contention â€” fall outside runbooks and require senior engineers.
Problem 4:
 No institutional memory â€” Every time an incident is resolved, the knowledge of how it was fixed lives in one engineer's head or a Confluence page nobody reads. The next incident of the same type starts from zero.

3. System Architecture
3.1 High-Level Design
ASHIA is composed of two independent layers: the Target System (the infrastructure being monitored and healed) and the AIOps Control Plane (the agentic system that monitors and heals it). These two layers communicate exclusively through well-defined interfaces: the Prometheus HTTP API, the Loki HTTP API, the Jaeger HTTP API, and the Docker SDK.
Core architectural principle
The AIOps Control Plane never reaches directly into the target system's code. It only interacts through the same observability and control APIs that any operator would use. This means the system is portable â€” point it at any observable system and it works.

3.3 AIOps Control Plane â€” LangGraph State Machine
The control plane is a LangGraph directed graph. Each node is a specialized agent. Edges are conditional functions that route based on the current AgentState. The graph is persistent â€” state survives crashes via the PostgreSQL checkpointer.
Graph execution lifecycle:
Monitor Agent runs on a 30-second polling loop (LangGraph periodic trigger). Anomaly detected â†’ writes AlertEvent to AgentState â†’ routes to Root Cause Agent.
Root Cause Agent queries Loki, Jaeger, and Pinecone. Feeds context to GPT-4o with structured CoT prompt. Writes ranked Hypothesis list to state â†’ routes to Remediation Agent.
Remediation Agent generates fix options per hypothesis. Scores each for risk. Conditional edge: LOW risk â†’ execute immediately. MEDIUM or HIGH risk â†’ trigger HITL interrupt.
HITL Interrupt serializes state to PostgreSQL. Sends Slack notification. Graph pauses. On human approval â†’ resumes to Remediation Agent execution. On abort â†’ routes to cleanup.
Verifier Agent monitors recovery metrics for 60s. Recovered â†’ route to Learning Agent. Not recovered AND retries < 3 â†’ route back to Root Cause Agent with next hypothesis. Retries exhausted â†’ escalate.
Learning Agent embeds full incident postmortem into Pinecone. Updates incident record in PostgreSQL. Graph terminates.

4.1 Agent â€” Monitor Agent (detailed specification)
The Monitor Agent is the always-on sensor of the system. It runs on a configurable polling interval (default 30 seconds) and is the only entry point into the incident response pipeline.
Monitored Metrics (PromQL queries)
Metric Name
PromQL Expression
HTTP error rate
rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])
Request latency p95
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))
Memory usage %
container_memory_usage_bytes / container_spec_memory_limit_bytes * 100
CPU throttle rate
rate(container_cpu_cfs_throttled_seconds_total[5m])
DB connection pool usage
pg_stat_activity_count / pg_settings_max_connections * 100
Cache miss rate
rate(redis_keyspace_misses_total[5m]) / rate(redis_keyspace_hits_total[5m])
Pod restart count
increase(kube_pod_container_status_restarts_total[15m])
Disk I/O wait
rate(node_disk_io_time_seconds_total[5m])
Anomaly Detection Algorithm
Fetch 24-hour rolling window of each metric from Prometheus.
Compute rolling mean (mu) and standard deviation (sigma) using a 1-hour sliding window.
Current value is anomalous if: abs(current - mu) > 2.5 * sigma.
Consecutive anomalies required before firing: 3 (prevents transient spike false positives).
AlertEvent payload includes: service_name, metric_name, current_value, expected_range, severity (LOW/MEDIUM/HIGH/CRITICAL).
4.2 Agent â€” Root Cause Agent (detailed specification)
The most cognitively complex agent. It receives an AlertEvent and must produce a ranked list of hypotheses with confidence scores by synthesizing evidence from three independent data sources.
Context Assembly Pipeline
Query Loki for last 50 log lines from the affected service with LogQL filter: {service="order-service"} |= "ERROR" | json.
Query Jaeger for the last 20 trace spans from the affected service covering the anomaly window.
Query Pinecone for the 3 most semantically similar past incidents using the AlertEvent as the embedding query.
Compile all three sources into a structured context object (max 12,000 tokens to stay within GPT-4o context budget).
Submit to GPT-4o with chain-of-thought system prompt. Require structured JSON output: list of Hypothesis objects.
GPT-4o Prompt Structure
SYSTEM: You are a senior SRE engineer performing root cause analysis.
You will be given live telemetry from a microservices system.
Reason step by step. Produce a ranked JSON list of hypotheses.

Each hypothesis must contain:
  - hypothesis_id: str
  - description: str (what went wrong and why)
  - confidence: float (0.0-1.0)
  - evidence: list[str] (which log lines / traces / metrics support this)
  - suggested_fix_category: Literal['restart','scale','rollback','config','db','cache']

USER: Alert: {alert_json}
      Logs: {log_lines}
      Traces: {trace_spans}
      Similar past incidents: {past_incidents}

6. Functional Requirements
6.1 Core Requirements â€” P0 (Must Have)
FR-01: Real-Time Anomaly Detection
Monitor Agent polls all 8 defined PromQL metrics every 30 seconds.
Z-score algorithm fires AlertEvent only after 3 consecutive anomalous readings.
AlertEvent is enriched with: service name, metric name, current value, expected range, and CRITICAL/HIGH/MEDIUM/LOW severity.
Monitor Agent must handle Prometheus API unavailability gracefully (exponential backoff, alert on 3 consecutive failures).
FR-02: LLM-Based Root Cause Analysis
Root Cause Agent assembles context from Loki, Jaeger, and Pinecone before each LLM call.
Context must not exceed 12,000 tokens (enforced by a pre-call token budget check).
GPT-4o response must be parsed into a valid list of Hypothesis objects using Pydantic. Invalid responses trigger a retry (max 2 retries).
Hypothesis list must be ranked by confidence score (descending).
Minimum 1 hypothesis required to proceed. If 0 hypotheses are generated after 2 retries, the incident is escalated to human unconditionally.
FR-03: Risk-Tiered Autonomous Remediation
Every fix option must have a computed risk_score of LOW, MEDIUM, or HIGH based on: action type, service criticality, time of day, and retry count.
LOW risk actions execute automatically via Docker SDK with no human approval.
MEDIUM and HIGH risk actions trigger the HITL Supervisor interrupt.
Remediation Agent must support at minimum: container_restart, image_rollback, memory_limit_update, service_scale_up, cache_flush, db_connection_pool_reset.
All executed actions are written to AgentState execution_log with timestamp, action type, parameters, and result.
FR-04: Human-in-the-Loop Interrupt
LangGraph interrupt node is triggered for any MEDIUM or HIGH risk fix, or after 3 failed retry loops.
On interrupt: full AgentState is serialized to PostgreSQL, Slack notification is sent within 10 seconds.
Slack message must contain: incident summary, root cause hypothesis, proposed fix, risk score, and Approve/Override/Abort action buttons.
Human can: (a) approve proposed fix, (b) override with a custom instruction, (c) abort the incident session.
All HITL interactions are logged to the audit trail with user_id, timestamp, action, and reason.
FR-05: Self-Healing Retry Loop
If Verifier Agent confirms no recovery within 60 seconds, LangGraph routes back to Root Cause Agent with the next hypothesis from the ranked list.
retry_count is incremented in AgentState on each loop.
Maximum retries: 3. After 3 failed attempts, incident is unconditionally escalated to human regardless of risk score.
Each retry must use a different hypothesis. Root Cause Agent must not repeat a hypothesis that was already attempted.
FR-06: Continual Learning Memory
Learning Agent runs on every resolved or escalated incident.
Postmortem schema: incident_id, service, alert_signature, root_cause_confirmed, fix_applied, fix_outcome, time_to_recovery, retry_count, timestamp.
Postmortem is embedded using OpenAI text-embedding-3-small and upserted to Pinecone with full metadata.
Root Cause Agent queries Pinecone with the current alert signature before every LLM call, retrieving top-3 most similar past incidents.
System must demonstrably resolve a repeated fault faster on second occurrence due to memory retrieval (validated in testing).

8. Core Data Models
8.1 AIOpsState â€” LangGraph State Schema
The central TypedDict that flows through every node in the LangGraph graph. Every agent reads from and writes to specific fields. No agent may write to fields outside its defined output contract.
class AIOpsState(TypedDict):
    # Session metadata
    incident_id:      str                   # UUID, generated at graph start
    user_id:          str | None            # Set if HITL has been invoked
    created_at:       datetime

    # Monitor Agent output
    alert:            AlertEvent            # metric, value, threshold, severity, service
    raw_metrics:      dict[str, float]      # snapshot of all 8 metrics at alert time

    # Root Cause Agent inputs & output
    logs:             list[LogLine]         # last 50 lines from Loki
    traces:           list[TraceSpan]       # OTel spans from Jaeger
    past_incidents:   list[PastIncident]    # top-3 from Pinecone similarity search
    hypotheses:       list[Hypothesis]      # ranked by confidence, descending
    current_hypothesis_idx: int             # which hypothesis is currently being tried

    # Remediation Agent output
    fix_options:      list[FixOption]       # ranked by risk score
    selected_fix:     FixOption | None
    execution_log:    list[ActionResult]    # all Docker SDK calls made

    # Verifier Agent output
    recovery_confirmed: bool | None
    recovery_metrics:   dict[str, float]   # post-fix metric snapshot

    # Control flow
    retry_count:      int                  # incremented on each replan loop
    hitl_required:    bool
    hitl_response:    HumanDecision | None

    # Learning Agent output
    postmortem:       Postmortem | None

    # Final status
    status:           Literal['active', 'paused', 'resolved', 'escalated', 'failed']
    resolved_at:      datetime | None
    time_to_recovery: float | None          # seconds
    total_cost_usd:   float                 # sum of all LLM API costs

12. Fault Injection Suite
The fault injection suite is a first-class deliverable â€” not an afterthought. It must be repeatable, configurable, and produce realistic telemetry patterns that match production failure signatures.
Fault Type
What It Does & Expected System Response
memory_leak
Progressively allocates memory in Order Service until OOM kill. Expected: memory_usage metric breaches threshold â†’ Monitor fires CRITICAL alert â†’ Root Cause identifies OOM â†’ Remediation restarts container.
cpu_spike
Runs a CPU-bound loop in User Service for 120 seconds. Expected: CPU throttle rate spikes â†’ MEDIUM severity alert â†’ Root Cause identifies runaway process â†’ Remediation restarts service.
db_connection_exhaustion
Opens 100 simultaneous idle DB connections, exhausting the connection pool. Expected: pg_stat_activity_count spikes â†’ HIGH severity â†’ HITL triggered due to DB risk.
slow_query
Injects a non-indexed query running every 5 seconds against Order Service DB. Expected: request latency p95 degrades â†’ Root Cause identifies slow query â†’ Remediation suggests index creation (manual HITL).
redis_overflow
Fills Redis to 95% memory capacity. Expected: cache miss rate spikes â†’ Monitor fires alert â†’ Remediation flushes expired keys.
cascade_failure
Injects memory_leak on Order Service + db_connection_exhaustion simultaneously. Expected: multi-signal alert â†’ Root Cause must reason across both signals â†’ demonstrates complex reasoning capability.
inject_fault.py usage:
# Single fault
python scripts/inject_fault.py --type memory_leak --service order-service

# Fault with auto-recover (resets after N seconds without agent intervention)
python scripts/inject_fault.py --type cpu_spike --service user-service --duration 120

# Cascade failure (demo mode)
python scripts/inject_fault.py --type cascade_failure --delay 30

# List available fault types
python scripts/inject_fault.py --list

14. Risks & Mitigations
Risk
Mitigation Strategy
GPT-4o produces invalid JSON for Hypothesis output
Wrap every LLM call in a Pydantic validation + retry decorator. Max 2 retries with a simplified prompt. On failure after 2 retries, escalate to HITL unconditionally.
Docker SDK permissions â€” container restart requires elevated privileges
Run Remediation Agent with a scoped Docker socket mount (read-write only for specific container names). Document security implications in README. Never mount /var/run/docker.sock in production without review.
Prometheus / Loki / Jaeger unavailability breaks Monitor Agent
Implement health checks on all three data sources at startup. Monitor Agent uses exponential backoff (3 retries). After 3 consecutive failures, fires a MetaAlert to HITL about observability stack degradation.
Token budget overflow â€” telemetry context exceeds GPT-4o 128k limit
Enforced token budget check before every LLM call. Log lines truncated to last 50. Trace spans limited to 20. Pinecone results limited to top-3. Hard cap at 12,000 tokens with assertion.
Runaway retry loop â€” agent loops indefinitely
retry_count field in AgentState with hard limit of 3. After 3 retries, conditional edge unconditionally routes to HITL regardless of risk score. Watchdog timeout of 15 minutes per incident.
LLM cost overrun during development and testing
Use GPT-4o-mini for Monitor Agent and Verifier Agent (classification only). GPT-4o only for Root Cause Agent. Implement Redis response cache with 1-hour TTL for identical context payloads. Set hard monthly budget in OpenAI dashboard.
Pinecone cold start â€” first incident has no memory
Pre-populate Pinecone with 10 synthetic past incidents covering all 6 fault types during setup. Document this as the knowledge base initialization step in README.
Scope creep beyond 6-week timeline
P1 features (FR-07 through FR-11) are frozen until all P0 gate criteria pass Phase 4. Kubernetes extension is v1.1 only. No new fault types added after Phase 3.
